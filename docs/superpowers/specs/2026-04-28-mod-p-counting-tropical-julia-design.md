# Mod-P CountingTropical matmul: Julia GEMM interface

**Date:** 2026-04-28
**Status:** Design — pending user approval
**Branch:** `counting-tropical`

## Goal

Expose the existing CUDA AoS counting kernel as a standard Julia matmul interface that operates on `Matrix{CountingTropical{T, Mod{P}}}` for `T ∈ {Float32, Float64}`, both Max-plus and Min-plus directions, with the modulus `P` carried in the count element type. No CRT, no BigInt, no implicit ones-counts assumption: the user supplies `(value, count mod P)` per cell on input and gets `(value, count mod P)` per cell on output.

This is what `GenericTensorNetworks` / `TropicalNumbers.jl` users already do on CPU; the GPU path mirrors the same shape so existing call sites can swap implementations.

## Non-goals

- BigInt / multi-prime / CRT reconstruction. Callers who want exact integer counts compose at the Julia level (loop over primes, CRT-combine themselves), or use the existing `count_ground_states_gpu` (BigInt) / `count_ground_states_gpu_u64` (u64 fast-path) entry points.
- Integer scalar types `T` (`Int32`/`Int64`). The existing CUDA kernel only instantiates `f32` and `f64`. Integer T is Tier-4 in the project roadmap; out of scope here.
- Chained matmul on-device. Each call uploads inputs and downloads outputs; round-trips for sequential operations.
- Type piracy on `Base.*`. The new path is exposed only via named functions; users opt in.

## Architecture

Four layers, communicating through plain numeric pointers:

```
Julia user API
    tropical_matmul(A, B)
    tropical_matmul_min(A, B)
    LinearAlgebra.mul!(C, A, B)
        ↓ split CountingTropical{T, Mod{P}} → (Vector{T}, Vector{Int32})
        ↓ row-major transpose (column-major Julia → row-major buffer)
Julia FFI (ccall)
    tg_matmul_mod_p_<T>_<dir>(a_val, a_cnt, m, k, b_val, b_cnt, n, p, out_val, out_cnt) -> c_int
        ↓
Rust C ABI (catch_unwind, errno-style return code)
    crates/tropical-gemm-cuda/src/c_api.rs additions
        ↓
Rust host driver
    crates/tropical-gemm-cuda/src/matmul_mod.rs (new)
    matmul_mod_p<T, D>(a_val, a_cnt, m, k, b_val, b_cnt, n, p, out_val, out_cnt)
        ↓ pack to PairF32/PairF64, upload, allocate, launch, copy back
Existing kernel
    counting_kernel::launch_counting_gemm<T, D>
    (AoS, Barrett reduction, naive/warpk shape-aware dispatch)
```

The kernel layer is unchanged. Only the host driver and FFI surface are new.

## Components

### Rust host driver (new file `matmul_mod.rs`)

```rust
use crate::{
    context::get_or_init_context,
    counting_kernel::{launch_counting_gemm, CountingCudaKernel},
    error::{CudaError, Result},
    gpu_mat::GpuMatrix,
    pair::PackPair,
};
use cudarc::driver::{DeviceRepr, ValidAsZeroBits};

pub fn matmul_mod_p<T, D>(
    a_val: &[T], a_cnt: &[i32], m: usize, k: usize,
    b_val: &[T], b_cnt: &[i32], n: usize,
    p: i32,
    out_val: &mut [T], out_cnt: &mut [i32],
) -> Result<()>
where
    T: PackPair + DeviceRepr + ValidAsZeroBits + Default + Copy + 'static,
    (T, D): CountingCudaKernel<T, D>,
{
    // 1. Validate: shape products, P range (1 < p < 2^31), buffer lengths.
    // 2. Pack (a_val, a_cnt) and (b_val, b_cnt) into Vec<T::Pair>.
    //    Reuse existing pair::pack_<T>(values, counts) helpers.
    // 3. ctx = get_or_init_context(0)?;
    // 4. Upload pair_a (M×K), pair_b (K×N) as GpuMatrix<T::Pair>.
    // 5. Allocate value_c (M×N, T) and count_c (M×N, i32) on device.
    // 6. launch_counting_gemm::<T, D>(ctx, &pair_a, &pair_b,
    //                                  &mut value_c, &mut count_c, p)?;
    // 7. Copy value_c, count_c back to out_val, out_cnt.
    Ok(())
}
```

The general AoS kernel handles arbitrary input counts (unlike the ones-specialized fast path), so this is exactly what we need. Naive vs warpk dispatch happens inside `launch_counting_gemm` based on shape — the warpk path here uses *non-transposed* B (the strided-B variant), since transposed-B is wired only into the ones-specialized launch. That is fine: the user-visible cost is a small slowdown for the M·N ≤ 64² && K ≥ 64 niche, not correctness.

### C ABI additions in `c_api.rs`

Four entries, generated by extending the existing macro pattern:

```rust
tg_matmul_mod_p_f32_max(...)  // returns c_int, 0 = ok
tg_matmul_mod_p_f32_min(...)
tg_matmul_mod_p_f64_max(...)
tg_matmul_mod_p_f64_min(...)
```

Each signature:

```c
int tg_matmul_mod_p_f32_max(
    const float*   a_val,
    const int32_t* a_cnt,
    size_t m, size_t k,
    const float*   b_val,
    const int32_t* b_cnt,
    size_t n,
    int32_t p,
    float*   out_val,
    int32_t* out_cnt);
```

Return codes (matching existing convention):
- `0` ok
- `1` invalid input (shape mismatch, P out of range, null pointer)
- `3` CUDA error
- `4` Rust panic across FFI boundary

`BoundTooLargeError` (code `2`) does not apply here — there is no CRT envelope. The Julia binding maps `2` defensively but never expects to see it.

All entries wrapped in `catch_unwind` for panic safety, identical to existing entries.

### Julia surface (in `CountingTropicalGEMM.jl`)

New deps in `Project.toml`: `TropicalNumbers`, `Mods`, `LinearAlgebra` (stdlib).

Public API:

```julia
"""
    tropical_matmul(A, B) -> C

Max-plus counting tropical matmul, mod P (carried in the count element type).
A and B must be `AbstractMatrix{<:CountingTropical{T, Mod{P}}}` with
`T ∈ {Float32, Float64}` and matching modulus P. Returns a fresh
`Matrix{CountingTropical{T, Mod{P}}}`.
"""
function tropical_matmul end

"""
    tropical_matmul_min(A, B) -> C

Min-plus variant. Operates on the min-plus counting tropical type.
"""
function tropical_matmul_min end

"""
    LinearAlgebra.mul!(C, A, B)

In-place version: writes the matmul into pre-allocated C.
Inferred direction from the eltype's tropical orientation.
"""
# overload of stdlib mul!
```

Internals (per direction):

1. Verify `eltype(A) == eltype(B) == eltype(C)`, all `CountingTropical{T, Mod{P}}` for the same `T` and `P`. Throw `ArgumentError` on mismatch.
2. Verify `1 < P < 2^31`. Throw `ArgumentError` otherwise.
3. Verify K-match. Throw `DimensionMismatch` otherwise.
4. Allocate `Vector{T}(undef, M*K)` and `Vector{Int32}(undef, M*K)` (and same for B), fill row-major from A: `a_val[i,j] = A[i,j].n`, `a_cnt[i,j] = Int32(A[i,j].c.val)` (extracting the residue from `Mod{P}`).
5. Allocate output buffers `out_val::Vector{T}`, `out_cnt::Vector{Int32}` of length `M*N`.
6. `_check_version()` (existing helper), then `ccall` to the matching `tg_matmul_mod_p_<T>_<dir>`.
7. On non-zero return code, call `_throw_for(code)` (existing helper, mapped to typed errors).
8. Re-zip output into `Matrix{CountingTropical{T, Mod{P}}}` (column-major target), constructing each cell as `CountingTropical{T, Mod{P}}(out_val[i,j], Mod{P}(out_cnt[i,j]))`.

For `mul!`, step 5 is replaced by writing directly into `parent(C)` via the same re-zip routine; the C buffer is reused.

### Min-plus type

`TropicalNumbers.jl`'s `CountingTropical{T, CT}` is max-plus. To avoid coupling to an upstream type that may or may not exist, this package defines its own min-plus counterpart:

```julia
struct CountingTropicalMin{T, CT}
    n::T
    c::CT
end
```

with `+` (semiring add: take the smaller `n`, sum counts on tie) and `*` (semiring mul: add `n`, multiply counts) defined to make this a tropical-min counting semiring. The Julia API dispatches on `CountingTropical` → Max C ABI, `CountingTropicalMin` → Min C ABI. Both types share marshalling code through a small internal helper that takes the eltype and direction symbol.

## Data flow (worked example)

User calls `tropical_matmul(A, B)` with `A :: Matrix{CountingTropical{Float32, Mod{7}}}` of size 4×6, `B` of size 6×3.

1. Julia wrapper extracts:
   - `a_val :: Vector{Float32}` of length 24, row-major (transpose of `[a.n for a in A]`).
   - `a_cnt :: Vector{Int32}` of length 24, row-major, each entry `mod(A[i,j].c.val, 7)` (already in [0,7) by Mod{P} invariant; cast to Int32).
   - Similarly for B (length 18).
   - `out_val :: Vector{Float32}` of length 12, undef.
   - `out_cnt :: Vector{Int32}` of length 12, undef.
2. `ccall(:tg_matmul_mod_p_f32_max, ...)` with `p = Int32(7)`, `m=4, k=6, n=3`.
3. Rust C ABI calls `matmul_mod_p::<f32, Max>` driver.
4. Driver packs `a_val/a_cnt` → `Vec<PairF32>` (8 bytes per element), uploads as `GpuMatrix<PairF32>` (4×6); same for B (6×3).
5. Driver allocates device `value_c: GpuMatrix<f32>` and `count_c: GpuMatrix<i32>`, both 4×3.
6. `launch_counting_gemm::<f32, Max>(ctx, &pair_a, &pair_b, &mut value_c, &mut count_c, 7)` dispatches naive (M·N=12 ≤ 64²; K=6 < 64 so warpk threshold not met → naive path).
7. Driver copies `value_c` → `out_val`, `count_c` → `out_cnt`.
8. Julia wrapper builds output `Matrix{CountingTropical{Float32, Mod{7}}}` (4×3) by zipping `out_val[i,j]` with `Mod{7}(out_cnt[i,j])`.

## Error handling

| Condition | Where caught | Surface |
|---|---|---|
| Shape K mismatch | Julia | `DimensionMismatch` |
| P ≤ 1 or P ≥ 2^31 | Julia (and Rust as defense) | `ArgumentError` |
| Mismatched eltype A/B/C | Julia (compile-time via dispatch where possible, runtime otherwise) | `MethodError` or `ArgumentError` |
| Output buffer wrong size (mul!) | Julia | `DimensionMismatch` |
| CUDA OOM / launch failure | Rust driver returns `Err(...)` | C ABI returns 3, Julia throws `CountingTropicalGEMMError(3, msg)` |
| Internal panic | C ABI `catch_unwind` | returns 4, Julia throws `CountingTropicalGEMMError(4, "Rust panic ...")` |

## Testing

New testset(s) in `CountingTropicalGEMM.jl/test/runtests.jl`:

1. **Reference equivalence (per direction × per T):** small random `A`, `B` over `CountingTropical{T, Mod{P}}` with small `P` (e.g. `P=7`) to guarantee residue collisions, K large enough to force ties (e.g. K=20). Reference: pure-Julia `for` loop performing max-plus + count-multiply mod P. Assert GPU result equals reference cell-by-cell (both `.n` and `.c.val`).
2. **Min direction:** symmetric to (1) with `CountingTropicalMin`.
3. **Edge cases:** P=2 (smallest valid), P=2^31-1 (largest valid prime fitting i32), K=1, 1×1×1 shape, all-tie input (counts compose).
4. **`mul!` reuse:** call `mul!(C, A, B)` twice with different `A`/`B`; assert C overwritten correctly each time.
5. **Errors:** P=1 → `ArgumentError`; P=2^31 → `ArgumentError`; K-mismatch → `DimensionMismatch`; mismatched moduli (`Mod{7}` × `Mod{11}`) → method/argument error.
6. **Existing tests** (u64 fast-path, etc.) remain untouched and must still pass — 17/17 → 17/17 + new tests.

## Performance expectations

The AoS general kernel (Spec F) hit ~666 G tropical-ops/s at 4096² square on A100 for f32/Max with single-prime modulus. That number is what users should expect from `tropical_matmul` on similar shapes. Marshalling overhead (split + zip in Julia) is O(M·K + K·N + M·N) cell-wise copies — orders of magnitude below kernel time at any non-trivial size.

The warpk path here is the *non-transposed* B variant, slower than the transposed-B specialization the ones path uses. If users hit the small-M·N high-K niche frequently and care about latency, that's a follow-up (parallel transposed-B AoS general kernel), out of scope for this spec.

## Open follow-ups (explicitly deferred)

1. Reusable scratch buffers (`MatmulModScratch`) for hot loops avoiding per-call device alloc.
2. Transposed-B AoS general kernel for the small-shape regime parity with ones path.
3. Integer scalar `T`.
4. On-device chained matmul (keep residues on device across calls).
